How to Use the Firmware Checker

üöÄ Quick Start

The Firmware Checker is a comprehensive platform for managing and verifying firmware versions across your datacenter infrastructure. It includes systems management, rack organization, firmware checking, and compliance tracking.

Main Features:
  • Systems Management: Track and organize all your systems by rack, room, and building
  • Rack Management: Organize racks by location, building, and room with RSCM component tracking
  • Firmware Checking: Verify firmware versions for DC-SCM, OVL2, storage, and drivers
  • Programs: Separate systems by program/project for better organization
  • User Management: Role-based access control (Admin, Editor, Viewer)

View Modes
  • Card View: Visual cards showing system overview
  • Table View: Compact table with all system details
  • Tree View: Hierarchical view by Location ‚Üí Building ‚Üí Room ‚Üí Rack ‚Üí U-Height ‚Üí Systems
  • Rack View: Grouped by rack with U-height organization
System Features:
  • Track system hostname, RSCM IP/port, rack assignment, and U-height
  • View firmware check history for each system
  • Filter systems by program
  • Bulk firmware checks on multiple systems
  • Edit system metadata (rack, room, location, U-height)

üì¶ Rack Management

Rack Organization

Racks are organized in a hierarchical structure:

  • Location: Geographic location (e.g., Redmond)
  • Building: Building identifier (e.g., Building 50)
  • Room: Room number (e.g., Room 1157, Room 1103)
  • Rack: Individual rack or bench name
Rack Features:
  • RSCM Components: Each rack tracks Upper and Lower RSCM IP addresses
  • Rack Types: Distinguish between racks and benches
  • System Count: See how many systems are in each rack
  • Tree View: Collapsible hierarchy showing location ‚Üí building ‚Üí room ‚Üí racks
  • Edit Racks: Update rack location, room, type, and RSCM IPs
  • Delete Empty Racks: Remove racks with no systems assigned
RSCM Component Management

RSCM (Remote Server Control Module) components are tracked per rack:

  • Upper RSCM: Primary management interface
  • Lower RSCM: Secondary management interface
  • Systems automatically associate with the correct RSCM based on their IP address
  • Bulk firmware checks use rack RSCM associations

üìÅ Programs

Program-Based Organization

Programs allow you to separate systems by project or team:

  • Assign systems to different programs
  • Filter views by program (affects Systems, Racks, and Firmware Check History)
  • Switch between programs using the dropdown in the navigation bar
  • View all systems across all programs or focus on a specific program

‚ûï Adding New Systems

Two Ways to Add Systems:
1. Manual Entry
  • Click "Add System" from Systems page
  • Enter system name, RSCM IP, port, and description
  • Assign to a program
  • Optionally assign to rack and U-height
2. Discovery & Metadata Entry
  • Run a firmware check on an unknown system
  • System is discovered with its serial number
  • Add metadata: hostname, rack selection, U-height, room
  • Create new racks on-the-fly with location and RSCM IPs
  • System is automatically linked to rack and RSCM components

üîç Firmware Checking

üîë Required Credentials

RSCM Credentials (Required)

Used for: DC-SCM and OVL2 firmware checks via Redfish API and SSH

  • Username: Typically "root"
  • Password: Your RSCM root password
Windows OS Credentials (Optional)

Used for: M.2/E.1s storage firmware and MANA driver checks

  • Computer Name: Target Windows computer name or IP (e.g., SERVER01 or 192.168.1.100)
  • Username: Windows administrator account
  • Password: Windows administrator password

Note: If OS credentials are not provided, storage and driver checks will be skipped.

üì¶ Firmware Categories

DC-SCM (Data Center Server Control Module)

Checks firmware for server management components:

  • BMC (Baseboard Management Controller)
  • BIOS
  • CPLD (Complex Programmable Logic Device)
  • Various configuration firmware
OVL2 (Overlake Card)

Checks firmware for FPGA and offload components:

  • FPGA Agilex Application Image
  • Cyclone V Image and PFMID
  • SOC Test OS (STOS)
  • SOC FIP and drivers
  • MANA Driver (Windows) - requires OS credentials
  • Glacier Cerberus firmware
Other Platform

Checks additional platform components:

  • HPM CPLD
  • SOC VR Configs
  • M.2 Storage - requires OS credentials (shows drive number and version)
  • E.1s Storage - requires OS credentials (shows all drives with versions)

üìù Step-by-Step Guide

1 Navigate to Check Firmware

Click "Check Firmware" from the navigation menu, or use "Check Firmware" button on a system's detail page.

2 Select or Enter System Information

Choose an existing system from the dropdown or enter new system details:

  • Existing System: Select from dropdown to auto-fill RSCM IP and port
  • New System: Enter RSCM IP address and port number
3 Provide RSCM Credentials

Enter your RSCM username and password (typically root/password).

4 Add OS Credentials (Optional)

If you want to check storage (M.2/E.1s) and MANA driver firmware:

  • Enter the target Windows computer name or IP
  • Provide Windows administrator credentials
  • Ensure WinRM is enabled on the target computer
5 Start the Check

Click "Start Firmware Check" button. The check will run in the background.

6 Monitor Progress

You'll be redirected to a progress page showing real-time updates as each firmware component is checked.

7 View Results

Once complete, you'll see all firmware versions organized by category. Green indicates success, yellow indicates warnings, and red indicates errors.

üìä Understanding Results

Status Indicators:
  • Success - Firmware version retrieved successfully
  • Not Found - Component not detected or not applicable
  • Not Checked - Skipped (usually due to missing credentials)
  • Error - Connection or execution error
Storage Firmware Display:

M.2: Shows as "Drive X: VERSION" (e.g., "Drive 4: LDBD1M4Q")

E.1s: Shows multiple drives separated by " | " (e.g., "Drive 1: 61080A50 | Drive 2: 61080A50 | Drive 3: 61080A50")

Each drive number corresponds to its physical disk number in the system.

üîß Troubleshooting

Common Issues:
Connection Failed / Timeout Errors:
  • Verify the RSCM IP address is correct
  • Ensure the server is powered on and accessible
  • Check network connectivity
  • Verify RSCM credentials are correct
Storage/Driver Checks Show "NOT_CHECKED":
  • Ensure OS credentials were provided
  • Verify the computer name or IP is correct
  • Confirm WinRM is enabled on the target computer
  • Check that the Windows account has administrator privileges
SSH Authentication Failed:
  • Verify RSCM username and password
  • Check if SSH is enabled on the RSCM
  • Ensure port 22 is accessible
Check Takes Longer Than 10 Minutes:
  • The system will timeout after 10 minutes
  • Check console output for specific component delays
  • SSH serial sessions may take longer on slow connections
  • You can retry individual components if needed

üìã Data Organization Best Practices

System Naming Convention

Use descriptive names that include:

  • Serial numbers for unique identification
  • Node identifiers (N1, N2) for blade systems
  • Example: P588853840044039-N2
Hostname Format

Standard hostname format: C41431157B0234B

  • C4143: Building/site identifier
  • 1157: Room number (automatically extracted)
  • B02: Rack identifier
  • 34B: U-height position
Rack Naming
  • Use clear rack identifiers: B02-Rack, E08-Rack
  • Distinguish benches: TestBench-5
  • Include rack suffix for clarity
Location Format

Standard format: City - Building Number

  • Example: Redmond - Building 50
  • Consistent formatting enables proper grouping in tree views

WinRM Requirements for Storage/Driver Checks:

To enable storage and MANA driver checks, WinRM must be configured on target Windows computers:

# Enable WinRM (run in PowerShell as Administrator)
Enable-PSRemoting -Force
Set-Item WSMan:\localhost\Client\TrustedHosts -Value "*" -Force
Restart-Service WinRM
Viewing History:

From the home page, you can:

  • View all previous firmware checks
  • Filter by system name, date, or program
  • Click any check to view detailed results
  • Compare firmware versions across different checks
  • See check status and duration
Tree View Navigation:

Both Systems and Racks pages support hierarchical tree views:

  • Click headers to expand/collapse sections
  • Chevron icons indicate expandable sections
  • Badges show system/rack counts at each level
  • Sections default to collapsed for better performance
  • State is preserved in browser session
Program Filtering:

Use the program selector in the navigation bar to:

  • View systems from a specific program only
  • Filter racks to show only those with systems in selected program
  • Focus firmware check history on one program
  • Select "All Programs" to see everything
Bulk Operations:

Perform actions on multiple systems at once:

  • Bulk Firmware Checks: Select multiple systems and run checks simultaneously
  • Select All: Use checkboxes to select all systems in a view
  • Group Selection: Select all systems in a rack or U-height group
  • Progress tracked independently for each system
